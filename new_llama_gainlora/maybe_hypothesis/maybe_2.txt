  BUG 2 (NGHIÊM TRỌNG): srt_max_emb_samples=200 vs kiểm định dùng TOÀN BỘ train data

  Đây là bug lý thuyết chính.

  - Kiểm định (routing_class_v2.py): Dùng TOÀN BỘ train embeddings (160, 1000, 1000... samples mỗi task). Tổng pool lên tới 11,741 samples.
  - Thực tế (run_superni_order1_llama_cl.sh line 49): SRT_MAX_EMB_SAMPLES=200. Mỗi task chỉ dùng tối đa 200 samples để tính centroid và covariance.

  Hệ quả:
  1. Centroid μ_t bị estimate kém: Với d=4096 và chỉ 200 samples, centroid estimate có variance cao → routing accuracy giảm
  2. Pooled covariance Σ_pool bị estimate cực kém: Với n/d = 200/4096 ≈ 0.049, ma trận covariance 4096×4096 được estimate từ chỉ 200 samples →
  rank-deficient nghiêm trọng. Ridge shrinkage δ = d/(n+d) = 4096/4296 ≈ 0.95 (gần như toàn bộ là identity matrix, mất hết thông tin anisotropic)
  3. So sánh: Trong kiểm định, sau 3 tasks, pool=2160, δ = 4096/6256 ≈ 0.65 → giữ được nhiều thông tin covariance hơn rất nhiều

  Nhìn vào log thực tế:
  - Task 1: n=160, δ=0.9624 (gần identity)
  - Task 2: n_pool=360, δ=0.9192
  - Task 3: n_pool=560, δ=0.8797

  Trong khi kiểm định:
  - Task 3: pool=2160 → δ ≈ 0.65

  Kết luận: Router trong thực tế hoạt động gần như NearestCentroid (L2 thuần) thay vì Pooled Mahalanobis thực sự, vì shrinkage quá cao khiến Σ_pool⁻¹ ≈
  identity.

  BUG 4 (QUAN TRỌNG): task_id_to_idx mapping bị đảo ngược thứ tự LoRA slots

  Trong sgwi_trainer_llama.py:179-182:
  task_id_to_idx = {current_task: 0}
  for prev_idx in range(self.cur_task_id):
      prev_task = self.task_order[prev_idx]
      task_id_to_idx[prev_task] = self.cur_task_id - prev_idx

  Ví dụ tại Task 3 (task1290_xsum):
  - task1290_xsum → slot 0 (current)
  - task1572_samsum → slot 2 (oldest, cur_task_id - 0 = 2)
  - task363_sst2 → slot 1 (cur_task_id - 1 = 1)

  Nhưng trong run_llama_gainlora_cl.py, LoRA adapters được load theo thứ tự:
  [PREV-LORA] Loaded slot[0] ← task363_sst2 (mới nhất)
  [PREV-LORA] Loaded slot[1] ← task1572_samsum (cũ nhất)

  Xác nhận từ log (line 465-466):
  [PREV-LORA] Loaded slot[0] ← .../2-task363_sst2.../saved_weights
  [PREV-LORA] Loaded slot[1] ← .../1-task1572_samsum.../saved_weights

  Trong LlamaModel.forward(), slot mapping:
  - key_attention_weights[:, 0] → current task LoRA
  - key_attention_weights[:, 1] → previous_lora_weights[0] = task363_sst2
  - key_attention_weights[:, 2] → previous_lora_weights[1] = task1572_samsum

  Và task_id_to_idx:
  - task1572_samsum → 2 → slot 2 → previous_lora_weights[1] = task1572_samsum ✓
  - task363_sst2 → 1 → slot 1 → previous_lora_weights[0] = task363_sst2 ✓

  Mapping này có vẻ đúng nhưng phụ thuộc vào thứ tự load LoRA. Cần verify kỹ rằng previous_lora_path được truyền đúng thứ tự (oldest→newest hay
  newest→oldest).

  Nhìn vào bash script (line 158-161):
  PREVIOUS_LORA_PATHS="${PREVIOUS_LORA_PATHS},${CUR_SAVED}"
  → Paths được append theo thứ tự oldest→newest. Nhưng trong code load (log line 465-466), slot[0] nhận path cuối cùng (newest) và slot[1] nhận path đầu
  (oldest). Điều này cho thấy code load đảo ngược thứ tự.

  Nếu mapping bị sai, router trả về đúng task nhưng activate sai LoRA adapter → performance drop nghiêm trọng.
